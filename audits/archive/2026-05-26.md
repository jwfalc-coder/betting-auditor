### F001 — paper_tracker NameError crash at 1400: a silent trade-loss bug

This is the primary operational finding for May 26. The 1400 paper_tracker crash (`NameError: name 'mb_hard_blocked_markets' is not defined`) is a newly-introduced bug — not present in any prior log file reviewed. It first appeared in the 1400 run on May 26, which used Matchbook as the selected platform.

**The crash sequence:**
1. Analyzer finds 2 signals at 10pp+
2. Paper tracker starts `log_paper_trades()` → calls `_log_paper_trades_locked()`
3. Signal 1 (Winner back on Matchbook): the matchbook_no_backs check fires, prints "Skipping MB back (matchbook_no_backs): Winner", and correctly blocks the trade
4. The function continues to Signal 2, which reaches the `if mb_hard_blocked_markets:` conditional at line 318
5. NameError: `mb_hard_blocked_markets` is undefined — crash terminates the function
6. The outer routine reports EXIT_CODE:0 (the paper_tracker's failure exit code is not propagated)

**What was lost:** Signal 2 was never evaluated. Given the 1400 run's market selection (15 Matchbook markets: 7 Winner outrights, 3 Moneyline, 5 Match Odds/other) and the BF cross-ref log showing Spain (Match Odds, 6pp edge — below threshold), the second 10pp+ signal was most likely a Match Odds or Moneyline back/lay on Matchbook. If it was a Match Odds or Moneyline back at odds ≥2.0 (which MOD-117/118 allows), it would have been a valid paper trade. The signal was dropped without trace.

**Why 1800 and 2300 didn't crash:** Both runs selected Betfair as the platform. The `mb_hard_blocked_markets` variable is specific to Matchbook signal processing — the code path is not reached when the researcher's output contains only Betfair signals.

**Variable name etymology:** `mb_hard_blocked_markets` vs `mb_blocked_markets` suggests a recent code refactor, likely as part of MOD-117/118, which added new back bet types. The refactor may have introduced a new variable (`mb_hard_blocked_markets` for the harder block rules, vs `mb_blocked_markets` for the standard soft blocks) without initialising it in all execution paths. The fix is a one-line change.

**Risk going forward:** The next time the 1400 run (or any Matchbook-primary run) sees 2+ signals where the first is a blocked Winner back, this crash will repeat. Given the log history, this pattern occurs regularly (many prior runs show "Skipping MB back (matchbook_no_backs): Winner" alongside other signals).
